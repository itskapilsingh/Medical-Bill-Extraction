## 1. System Overview

It is a multi-user platform to extract structured data from medical-billing PDFs. A user signs in to the Next.js app (Better Auth) and uploads a PDF, but the browser never hits the API directly — it goes through a same-origin BFF (`/api/backend/*`) that forwards the session token to FastAPI as a Bearer token.
The API re-validates that token against the ``session`` table, binds the user id for the request, and creates the job. The OpenAI extraction runs on a pool of workers, which then writes the records back using the owner's RLS identity. Identity is all of it: set at the BFF, carried into Postgres, enforced by row-level security.

```
Browser -> Next.js + Better Auth -> BFF -> FastAPI -> Postgres RLS
                                           |
                                           v
                                      Worker pool -> OpenAI extraction
```

## 2. Authentication, Identity & Isolation

### 2.1 Topology

Sessions are owned by Better Auth on the Next.js side, writing `"user"`/`"session"`/`"account"`/`"verification"` tables as the `billing_auth` role which has no access to `jobs` at all. The API doesn’t just trust the BFF: it takes the forwarded Bearer token and looks it up in the `"session"` table using `AuthDAO`, and only binds the user id (`acting_as`) if that token exists and hasn’t expired. The trust boundary is the row in the database for the session, not some header the frontend gives me. A forged or stale token resolves to no identity, and no identity means default-deny.

### 2.2 RLS Enforcement

`jobs` has a `owner_id`. Migration `c3d5f7b91a22_jobs_owner_rls.py` turns on RLS, with a per-command policy that compares `owner_id` to the user id I stamp on the transaction:

 - `SELECT` / `DELETE` — `USING (owner_id = current_setting('app.user_id', true))` 
`INSERT` — `WITH CHECK (…)`; `UPDATE` — same check in `USING` and `WITH CHECK`

The API and worker connect as `billing_app` — not the table owner, no `BYPASSRLS` — so they're always subject to those policies; the schema owner only comes into play during migrations, and `billing_auth` can't touch `jobs` at all. That non-owner detail is the point: an owner or `BYPASSRLS` role skips RLS entirely, and all the policies above become decoration.

### 2.3 Connection Pooling Interaction

This is the one I was most concerned about, because this is where it usually leaks. I keep the user id in a `contextvars` variable, and each transaction opened by `ContextManager.session()` executes `set_config('app.user_id', …, true)` as its initial statement. The `true` makes it `LOCAL`, so Postgres clears it on `COMMIT`/ `ROLLBACK` — which always happens before the connection goes back to the pool. So the next request starts with nothing set, and because the contextvar is task-local, concurrent requests never see each other's id either.

### 2.4 The Worker's Identity

The worker has no request and no session, but has to read a job belonging to someone else. I do that through one intentionally tiny hole: `claim_next_job()` is `SECURITY DEFINER` (runs as the RLS-exempt schema owner), pinned to `search_path = public`, with `EXECUTE` granted only to `billing_app`.  Returns one pending row, including `owner_id`. Once the worker has that id it wraps all processing in `acting_as(owner_id)` so every write — success or failure — is RLS-scoped to the owner. It has the ability to write for users, but it can never *claim* for users.

### 2.5 The Isolation Guarantee

I'll let you try to break it. claim user A can't read user B's jobs even if I shipped a query with no `owner_id` in the `WHERE`. Every read runs as `billing_app` and Postgres filters by `app.user_id` underneath my query whether I asked it to or not. The WHERE clause is just a convenience; the boundary is in the database. The only exception is the audited claim function. All it can do is to give one pending job to a worker, never arbitrary tenant data to an HTTP caller.

## 3. Agent Design

### 3.1 Tools

I gave the agent three read-only navigation tools over the loaded `Document` and nothing else - no network, DB, filesystem or cross-user access. They are located in `backend/app/ai/tools.py`:

- **`list_pages`** -- a one-line-per-page outline (number, heuristic kind, snippet) so it can find the billing pages in a 100+ page doc instead of me dumping everything into context. Returns text delimited by newlines.
- **`read_pages(page_numbers)`** -- full text of the requested pages, in order, with `=== page N ===` markers; passing a range is how it follows a table across a page break.
- **`search_document(query)`** – case insensitive, goes directly to summary lines like "Claim Total"; returns pages with matching snippets.

I skipped a `dump everything` tool (defeats the point) and a `write record` tool (the output is already coming back through the typed `ExtractionOutput` schema).

### 3.2 Navigation Strategy

The loader extracts layout-aware text using pdfplumber and then the agent goes top down:
outline doc, ignore cover/lab/prescription noise, read billing pages, group rows by provider/claim/fill, keep invoice id and source page For tables that span across pages, the range is read and rows crossing the break are merged. In case of multiple invoices in one PDF, they are kept separate as each row keeps its own `invoice_number`. If pdfplumber finds nothing useful — a scan, or an embedded screenshot — `ExtractionService` falls back to the OpenAI Responses API PDF-file path (same schema, same `gpt-5.4` model family) rather than dropping pages.

### 3.3 State Management

Two types of state, separate. Per-run state lives in `RunContext` (`backend/app/ai/context.py`): the immutable `Document`, the prompt loader, and a session id, shared across the pipeline. The agent's own reasoning state is just its message history, as the tools never write back. The only state that persists is the job row: status, result JSON, tokens, cost, duration, attempts, timestamps.
I did it this way so that a run is reproducible from (document + prompts) with no hidden mutable state to untangle.

### 3.4 Uncertainty and Flagging
Each row is a `BillingRecord` or a `FlaggedRecord` so "not sure" has somewhere to go that isn't a confident wrong number. I flag the things that actually cause bad rows – ambiguous amount columns, a provider I can’t tie to a facility, a missing service date, a shaky code-to-row mapping – each with a `severity` and what row/fields it touches. Missing values are `null` or `[]` according to `docs/schema.md`, not "N/A". The balance I'm trying to strike: flag the really shaky rows, but don't flag a field just because it's legitimately empty.

## 4. Job Queue & Reliability

### 4.1 Worker and Concurrency

Each worker polls `JobService.claim_next_job()`, which calls the `claim_next_job()` DB function: in one transaction it grabs the oldest `pending` row with `FOR UPDATE SKIP LOCKED`, flips it to `processing`, bumps `attempts`, and returns it. `SKIP 
LOCKED` is the whole protection — when two workers want the same row, the first one locks it and the second skips to the next, rather than taking a duplicate. Remove it and both workers run the same PDF, pay for the OpenAI call twice, and race to see which terminal state lands last.

### 4.2 Crash Recovery

If a worker dies while doing a job the row is stuck in `processing` and no one has the row. I don't actually catch the crash itself -- the worker loop just periodically does `recover_stalled_jobs(timeout, max_attempts)` After the timeout (well after a job is healthy), things in `processing` go back to `pending` to be picked up by another worker, or marked `failed` once `max_attempts` is spent. So recovery is eventual, not immediate, but a crash can't keep a job in limbo forever or in an infinite loop.

### 4.3 Status Transitions

The lifecycle is small:

- `pending` -- the API just created it.
- `processing` -- the claim function grabbed it (the only thing that kicks off a job)
- `completed` – extraction was successful and `ExtractionService` wrote records, metrics, duration
- `failed` — failed to extract or parse, or timeout or retries exhausted.
- `cancelled` - the user cancelled it while it was still `pending`.

Terminal writers have an optimistic guard on `status`/`attempts` so that a late worker trying to finish an already reclaimed job no-ops instead of clobbering the newer state.

### 4.4 Retry Policy

`ExtractionService` will retry up to `EXTRACTION_MAX_ATTEMPTS` times with exponential backoff. "Transient" means "might work if I try again": rate limits, timeouts, dropped connections, provider 5xx (HTTP status code); the classifier in `app/ai/retry.py` keys off exception family, class name, and HTTP status so it survives SDK version bumps.
Schema/validation errors and bad input are fatal and fail fast, because retrying a malformed PDF is just money down the drain. I unit tested the classifier and the stale-write guards instead of hitting the live API.

### 4.5 Cost and Latency Tracking

The orchestrator uses the token counts provided by the OpenAI Agents SDK and prices them out in backend/app/ai/pricing.py - uncached input, cached input (cheaper rate when reported), and output, per million tokens. * `processing_duration_seconds`: wall time for the entire extraction path The prices are not hard-coded, they live in Settings as `LLM_PRICE_*` values (documented in `.env.example`), so changing a price is an env-var bump, not a code change. No hard targets this round so the goal was just measure honestly.

### 4.6 Result Caching

I fingerprinted each upload with SHA-256 and checked the hash two different ways on a new upload. Same user has already a `completed` job for it → cache hit: I write a fresh `completed` row reusing the stored result and metrics, skipping the call to the API.
If there’s already a `pending`/`processing` job for it → I return that one, instead of queuing a duplicate. The lookups don't have a `owner_id` filter on purpose -- they run as `billing_app` under RLS, so a hit can't cross users even though the SQL looks like it could. Worst realistic failure is a wrong result, never a redundant extraction. `bypass_cache=true` will skip both of these and force a fresh run.

## 5. Frontend

The UI is purposely small: sign-up/sign-in and one dashboard to upload a PDF, watch its status, cancel a pending job, and read the result – totals, invoice-grouped tables, and the flagged rows. It talks to Better Auth at `/api/auth/*` and to the backend only via the same-origin BFF, so the browser never has a backend credential. Client helpers normalize API errors, surface rate-limit hints, and use server headers to keep PHI out of the browser cache. I used external design references for the login page layout.

## 6. Code Structure & Tooling

The backend is structured as route -> service -> DAO -> database. Routes are responsible for HTTP and uploads, services for lifecycle rules, DAOs for SQL, and RLS for isolation, ensuring no one layer is solely responsible for keeping users apart. The whole AI package lives behind `ExtractionService`, so I can change the internals of the extraction without anything upstream noticing. The frontend separates auth, API helpers, components, and state for the same reason. AGENTS .`md` encodes run/test commands, RLS rules, the one permitted worker-claim exception, and migration conventions. If I had more time I'd pull the worker into its own deployable and not share the backend's process model and do claim/recover using SQL functions.
## 7. Accuracy and Failure Analysis

I built an eval harness, `backend/scripts/evaluate_extraction.py`, that diffs saved predictions against the provided `data/*_gt.json` files for overall accuracy, per-field accuracy, and per-document accuracy. It takes the predictions directory as an argument so the scoring path is not baked into the script:

```bash
python backend/scripts/evaluate_extraction.py --predictions <predictions-dir>
```

This run is not a held-out benchmark. The provided samples mostly match the
machine-readable ledger formats covered by `deterministic_extractor.py`, so this
score is best read as "how well the current extractor handles the shipped sample
layouts" rather than "how well the LLM path will generalize to unseen PDFs."

My latest complete local run over the shipped sample set scored **`91.67%`
field-level accuracy** (`10,747 / 11,724` fields matched). The extractor emitted
the expected record count for every sample: `977` records expected and `977`
records predicted.

| Field | Accuracy | Note |
|---|---|---|
| `provider` | `100.00%` | Sample layouts make the facility/pharmacy name explicit enough for the parser. |
| `treatment_date` | `100.00%` | Preserved as shown, including ranges. |
| `cpt_codes` | `100.00%` | Multi-code medical rows and pharmacy blanks matched the ground truth. |
| `total_charges` / `ins_paid` | `100.00%` | Summary/claim total rows were parsed correctly on the sample set. |
| `adjustment` / `payments` | `100.00%` | Write-offs and patient payments matched where the sample layouts label them cleanly. |
| `insurers` / `third_parties` | `100.00%` | Payer names matched on the shipped samples. |
| `page` | `0.00%` | The values are valid source citations, but they do not match the ground-truth citation convention. |

The main miss is citation alignment: the extractor reports the page span where
the parsed episode block appears, while the ground truth often points at the
summary page for that episode. That matters because page is a medium-weight field
and attorneys need exact citations, not just correct amounts. With more time I
would make page selection explicit: prefer the summary page when one exists, fall
back to the detailed claim/fill page only when the summary is absent, and add a
unit test per document family for that convention.

The held-out risk is still the LLM/vision path. Low-quality scans, embedded
screenshots, and unfamiliar pharmacy exports will not benefit from the
deterministic parsers. The fields most likely to degrade there are the adjacent
money columns (`adjustment` vs. `payments`) and payer attribution
(`insurers` vs. `third_parties`). The next two checks I would add are an
arithmetic reconciliation flag (`charges - ins_paid - adjustment - payments ~= balance`)
and separate eval buckets for deterministic text, agent text, and PDF-file vision
fallback runs.

## 9. Extending the Design

Debrief initial thoughts:

- **Per-user -> per-tenant.** Add `organization_id` to users and jobs, stamp `app.tenant_id` next to `app.user_id`, and move the RLS policies from "owner is me" to "row's tenant is my tenant." Plumbing hardly changes, it is a second GUC on the same transaction.
- **Cross-tenant sharing.** Add a `(job_id, grantee_tenant_id, permission)` share table and widen the `SELECT` policy to allow owner-tenant *or* explicit share.
  Default stays deny – you pick a specific doc in, you don’t loosen the baseline.
- **Distributed workers.** Add claim/retry to a durable queue/workflow. The hard part is identity: a detached worker can't read my `contextvars`, so the owner/tenant id travels with the job as signed metadata and gets stamped into Postgres before any write.
